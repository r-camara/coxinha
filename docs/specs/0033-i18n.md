# Spec 0033: Internationalization (i18n + l10n)

- **Status:** in-progress
- **Phase:** F1
- **Owner:** Rodolfo
- **Depends on:** —
- **Relevant ADRs:** ADR-0012

## Why
No UI string can be hardcoded in a component. Translators must be
able to add a locale by dropping a file, without touching code.

## Scope

### In
- `react-i18next` + `i18next` set up in the frontend
- `rust-i18n` set up in src-tauri
- Locale files:
  - `src/locales/en.json` — all frontend strings
  - `src-tauri/locales/en.yml` — all Rust (tray) strings
- Translation helper wrapper:
  - Frontend: `useTranslation()` hook, `t('key')`
  - Rust: `t!("key")` macro
- OS locale detection at startup; fallback to `en`
- "Language" setting in the Settings view (future UI; key exists in
  `AppConfig.locale`)

### Out
- Pluralization and ICU syntax beyond simple interpolation → F1.5
- RTL languages → F1.5
- Date/number/currency localization → deferred; use `Intl` ad hoc
  until real needs come up
- Additional locales beyond `en` → ship locale files as needed

## Behavior (acceptance)
- **No raw strings in JSX** (besides the brand "Coxinha"): every
  visible text resolves through `t(...)`
- **No raw strings in tray/menu Rust code**: resolved via `t!(...)`
- **Missing key:** logs a warning in dev, renders the key text in prod
  (not an empty string)
- **Unknown locale:** falls back to `en` without crashing
- **Adding a new locale** means dropping a file and registering it;
  no component changes needed
- **Lint/CI check** (future) catches hardcoded strings in `src/` and
  missing keys across locales

## Design notes
- `src/lib/i18n.ts` initializes i18next and imports `en.json`
- `src/main.tsx` imports `./lib/i18n` before the React tree mounts
- `src-tauri/src/lib.rs` uses `rust_i18n::i18n!("locales")` at crate
  root
- Keys grouped semantically: `nav.*`, `sidebar.*`, `empty.*`,
  `agenda.*`, `meetings.*`, `settings.*`, `tray.*`
- Locale kept in `AppConfig.locale` (string like `en`, `pt-BR`);
  persisted in `config.toml`

## Open questions
- Share the same catalog between Rust and frontend, or keep two?
  (Decision in ADR-0012: two. Revisit if key drift becomes a pain.)
- Extraction tooling: `i18next-parser` optional in CI to catch
  missing keys?

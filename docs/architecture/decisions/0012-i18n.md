# ADR-0012: i18n with react-i18next + rust-i18n

- **Date:** 2026-04-18
- **Status:** Accepted

## Context
The project's UI strings must not be hardcoded in components. Users
may run the app in any locale, and strings need to live in locale
files that a translator can edit without touching code. The tray
menu is built by Rust; the rest of the UI is built by React. We
need one approach per side.

## Decision
- **Frontend:** `react-i18next` (with `i18next` core). Locale JSON
  files under `src/locales/<lang>.json`. Components call
  `useTranslation()` and `t('key')`.
- **Rust:** `rust-i18n` crate. YAML locale files under
  `src-tauri/locales/<lang>.yml`. Code uses the `t!("key")` macro.
- **Default locale:** `en`. Fallback: `en`.
- **Locale selection:** derived from the OS at startup (future
  override via the Settings view).

## Consequences
- **+** Standard libraries in both ecosystems, well-documented
- **+** Strings live in files a translator can PR without touching code
- **+** Separate frontend/backend catalogs keep each side simple
- **+** `t('key.nested')` reads cleanly
- **−** Two catalogs to keep in sync when a key appears on both sides
  (tray label duplicated in frontend — rare)
- **−** Extra runtime cost in frontend bundle (~30KB for
  react-i18next + i18next)
